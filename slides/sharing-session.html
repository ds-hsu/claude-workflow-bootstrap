Claude Workflow Bootstrap

用 AI 對話建立團隊知識工作流

分享人：許毓安 / ds-hsu

github.com/ds-hsu/claude-workflow-bootstrap

開場 30 秒。今天分享一個我整理出來的 bootstrap，目的是讓團隊新人能用最少步驟跟 Claude 一起工作。

我們遇過的痛

  • 新人裝環境痛苦（裝 Git → Node → SDK → IDE → ... 一小時起跳）
  • 腳本破在 Mac（PowerShell only，Mac 同仁怎麼辦？）
  • 文件總是過時（人類沒時間維護 wiki）
  • credentials 不小心進 commit
  • AI 工具用了一陣子，還是不知道它的能耐
跟現場有共鳴後再進解法。讓他們點頭：「對對對，這就是我的問題」。

解法

裝兩個工具之後，
其餘跟 AI 對話就好。

不是「AI 幫你寫 code」、是「AI 是工作流的一部分」

這是整場的核心觀念。從「跑腳本」轉成「跟 AI 講話」。

Part 1

AI 的核心能力

—— 兩大基石（Agents + Tools）與五個你可能還沒體驗過的能力

基石 #1 — Agents（分身）

傳統 AI 對話 = 你一句它一句、單線完成。

Claude 工作流 = 你一句、它派一隊分身平行做事。

內建 agent 類型用途
Explore讀檔搜尋 / 找 code（context 隔離、不污染主對話）
Plan實作前先寫計畫、評估架構取捨
code-reviewer獨立看 diff、給第二意見
feature-dev:code-explorer分析既有功能、追執行路徑、map 架構
general-purpose複雜多步驟任務、需要工具完整存取

範例：「用 Explore 找出哪些檔案會被 /ingest 觸及」→ 子 agent 平行回報，主對話 context 乾淨。

強調：context 隔離很關鍵。讓你不必把長串搜尋結果塞進主對話。

基石 #2 — Tools（直接操作）

Claude 不只是給建議讓你執行，它自己執行、自己驗證、自己迭代：

檔案系統

  • Read / Write / Edit
  • Glob（找檔）/ Grep（找字串）
  • NotebookEdit（Jupyter）

執行

  • Bash（任意指令、背景執行）
  • TaskCreate / TaskUpdate

網路

  • WebFetch / WebSearch

MCP 工具（外部整合）

  • Playwright（瀏覽器自動化）
  • Firecrawl（爬蟲）
  • Gmail / Calendar / Drive
  • claude-mem（跨 session 記憶）

Agents + Tools = Claude 能做事的兩大基石。下面 5 張 slide 是它們組合出來的具體表現。

這頁鋪墊「為什麼 Claude 跟 ChatGPT 體感差很多」。後續 capability slides 都是 agents + tools 的展示。

表現 #1 — 跨平台Tools 應用

傳統做法：


# Windows
.\scripts\setup.ps1

# Mac
./scripts/setup.sh

# Linux
./scripts/setup-linux.sh

AI 做法：


你（Win 或 Mac，不重要）：
  「幫我設定 ~/.claude/」

Claude 自己判斷 OS、處理路徑、merge JSON

沒有雙寫成本。Claude 自己解決平台差異。

強調：以前我們維護 .ps1 + .sh 雙份腳本，每改一處改兩次。現在不用了。

表現 #2 — 文件理解Tools + Agents

把一份 raw 文件（PDF / Word / drawio / md）丟進 raw/


你：「ingest 未處理的 raw」

Claude 做的事：

  • 讀文件、抽取系統 / 流程 / 實體 / 故障案例
  • 更新 5-15 個相關 wiki 頁面（不只 1 個）
  • 建立雙向 wikilink（避免孤立頁）
  • raw 與既有 wiki 衝突時標 ⚠️ CONFLICT、不私自決定

→ 「LLM 維護式知識庫」：文件不再過時，是因為 AI 不會疲倦

幕後：主 agent 讀 raw，可派 Explore agent 比對既有 wiki 找衝突，最後再用 Write/Edit 工具批次更新

這是專案的核心：Karpathy LLM Wiki 模式。AI 把「維護 wiki」這個傳統高成本工作變成幾乎零成本。Agents + Tools 都用上了。

表現 #3 — 知識管理 / Gap 偵測Tools

你問：「ShopDemo 退款流程涉及哪些系統？」

Claude 沒查到答案時，不會靜默失敗：


### gap-2026-05-13-001
- aspect: ShopDemo 退款流程系統涉及範圍
- status: open
- severity: medium
- created: 2026-05-13
- query_text: "ShopDemo 退款流程涉及哪些系統？"
- search_trail: [INDEX_LOOKUP:Procedures→miss,
                 COVERAGE_LOOKUP:refund-flow→miss,
                 RAW_FALLBACK:raw/.../退款.md→partial]
- resolved_by: null

下次有人補了資料 → /ingest 自動切 resolved_pending → 下次查詢驗證可回答 → 切 resolved

→ 知識斷層自動追蹤、生命週期化

這個機制叫 Coverage State / Gap Lifecycle。設計重點：兩階段驗證——AI 不能自己關閉 gap，要靠下次實際查詢驗證。避免 AI 自己騙自己。

表現 #4 — 失敗自我復原Tools

傳統腳本失敗：

  • 看 stack trace
  • Google 錯誤訊息
  • 改 script、再跑
  • 跑到一半又掛...

AI 工作流失敗：


你：「剛剛 /init-product-wiki 跑到一半噴錯，
     幫我看一下狀態」

Claude：
  - 用 Bash 工具掃目錄當前狀態
  - 判斷是該回退、繼續、或重來
  - 告訴你下一步怎麼做
很重要的能力。「跑壞了」變成可對話的問題，不是孤立 debug。

表現 #5 — 行為自律兩者 + 護欄

你不可能光靠運氣不出事。三層護欄 + AI 自律：

層機制例
1. 機械 deny絕對阻擋sudo / rm -rf / git push --force
2. 機械 ask二次確認psql / docker exec / npm install -g
3. AI 自律主動報告「我接下來要連 db-prod-...（看起來是 prod），跑 SELECT ...，要繼續嗎？」
4. 人類把關最終決策你看到上面那句，回 Y or N

每次連 DB 都被攔——刻意設計，把「連 DB」這件事拉到意識前緣。

強調：AI 自律是第三層、不是唯一層。每層都缺一不可。

Part 2

從零開始

—— 一杯咖啡時間從裝環境到跑出 BDD 規格

動工前先記住：Git 約定團隊及營運流程

後續 Claude 會幫你 commit / push / 開 PR——但要符合團隊規範。先過目這幾條：

Branch 規矩

  • 永遠從 main 開分支
  • 命名：feature/{username}/{功能名}
  • 例：feature/ds-hsu/bankInfo

Commit Message 格式

<type>(<scope>): <subject> (#issue)
  • type：feat / fix / docs / refactor / chore…
  • subject ≤ 50 字、直述句、命令句
  • 一個 commit 只做一件事

範例：


feat(對講機): 新增對講機部分授權功能 (#69437)
fix(社區公告): 把 tinymce 改成本地檔案，修正錯誤提示 (#70853)
refactor(手機驗證): 把手機驗證抽換成共用函式 (#87233)
docs: 社區選擇元件新增註解

這些約定寫進團隊 ~/.claude/rules/team-rules.md，/setup-claude 之後 Claude 跑 /commit 會自動套用。你只需要記得這格式存在、看 commit 訊息能驗 AI 對不對。

強調順序：先講 git 約定，再裝工具。理由：之後 Claude 跑 /commit 就是依這格式產出，學員看得懂才能驗證 AI 對不對。

Step 1 — 裝兩個工具3 分鐘

Windows（PowerShell）：


# Git：版本控制（剛剛那些約定的執行載體）
winget install --id Git.Git --silent --accept-package-agreements

# Claude Code：本場主角
winget install --id Anthropic.ClaudeCode --silent --accept-package-agreements

macOS（Terminal）：


brew install git                              # 版本控制
brew install anthropic/anthropic/claude-code  # 本場主角

關閉視窗重開後驗證：git --version + claude --version 都有版本回應即成功

→ 整場唯一需要手動敲指令的部分

Git 放第一行不是巧合——後續所有工作流（包含 Claude 自己）都依賴 git。

Step 2-4 — 跟 Claude 對話15 分鐘

Clone repo → cd → claude → 然後：

步驟你說的話Claude 做什麼
2 「幫我設定 ~/.claude/」 merge 安全規則進你的設定（17 deny + 17 ask）
3 「幫我建立 MyApp 產品 wiki」 互動問代號 / 路徑 → 建骨架 + 4 個起手 skill + git init
4 「跑電商範例 walkthrough」 建拋棄式 demo-wiki + 投放假資料 + 引導你跑 5 個步驟

每個 skill 都會在做動作前先告訴你「我打算改這幾個檔」，你隨時打斷、調整、重來。

團隊及營運流程 寫完 code 跟 Claude 說「/commit」或「commit + push + 開 PR」：Claude 自動套上前頁的 commit 格式，符合團隊規範。

這就是 bootstrap 的核心：4 步驟、3 步是對話、不寫腳本。Commit 時 Claude 也依團隊規範。

一杯咖啡時間長什麼樣？

┌─────────────────────────────────────────────────────────────────┐ │ │ │ T+0min 開 Terminal、winget/brew install ......................│ │ T+3min git clone bootstrap、cd 進來、claude .................│ │ T+5min 「幫我設定 ~/.claude/」.............................. │ │ T+7min 「幫我建 MyApp wiki」................................│ │ T+10min cd MyApp-wiki、開 Claude ............................│ │ T+12min 把 4 份需求文件丟進 raw/ ............................│ │ T+15min 「/ingest」 → 結構化 wiki 出現 ......................│ │ T+18min 「/wiki-query 'XXX 系統怎麼運作？'」.................│ │ T+20min 「規劃功能 Y」(自然語言) ............................│ │ T+25min 「/spec-by-example Y」→ BDD 規格 ....................│ │ │ │ ☕ 你的咖啡還是熱的 │ │ │ └─────────────────────────────────────────────────────────────────┘
這個 timeline 是真實的。我自己重跑了幾次計時。

Part 3

日常工作流

—— bootstrap 設定完之後，每天都怎麼用？

三層架構：Raw / Wiki / Schema

┌──────────────────────────────────────────────────────────────────┐ │ raw/ ........... 你投放的事實（PDF、Word、筆記，不可變） │ │ ↓ │ │ /ingest ←── 由 Claude 維護 │ │ ↓ │ │ wiki/ .......... LLM 結構化的編譯產物（會跟著 raw 更新） │ │ ↑ │ │ /wiki-query ←── 查詢 + 偵測 gap │ │ │ │ CLAUDE.md ...... Schema（規則與品質規範，人類維護） │ └──────────────────────────────────────────────────────────────────┘
  • raw 不可變：任何衝突以 raw 為準
  • wiki 唯一寫入入口是 /ingest：其他 skill 禁止直接寫
  • Schema 演化緩慢：CLAUDE.md 是規則本身，由 maintainer 改
這套架構讓「文件複利成長」——每次 ingest 都讓 wiki 更完整、不會 entropy。

個人工作流：累積 + 對外彙報個人工作流

兩個 skill 同名「daily」但目的不同——一個給未來的你，一個給主管：

/daily-log /daily-report
對誰：未來的自己（細粒度） 對誰：主管 / 團隊（粗粒度）
寫到哪：
raw/daily/{user}/{yyyyMMdd}.md		 寫到哪：
raw/daily-reports/{user}/{yyyyMMdd}.md
內容：bug 根因、踩到的坑、發現的系統隱含邏輯、知識庫更新 結構：固定四區塊（日常工作 / 中長期 / 團隊及營運流程 / 自我成長）
自動補齊：掃今日所有 session transcript，沒記錄到的補上 冪等：同日再呼叫會合併新項目進既有區塊

完整工作鏈：

完成工作 → /daily-log（個人細節累積） ──┐ ↓ → /daily-report（主管視角彙報） │ ↓ → /ingest（系統知識沉澱進 wiki） ──┤ ↓ → /insights-retro（週末 retro 洞見） ──┘

→ 個人筆記不再寫了就忘；它是團隊 wiki 的種子

daily-log 適合愛記筆記的人；daily-report 適合公司要求每日匯報的人。兩者並行不衝突。 強調：daily-log 內容會被 /ingest 後續收割進 wiki，所以「寫日誌」不再是孤立行為。

SBE — 寫 code 前先寫例子需求理解優先

Specification by Example：用具體例子釐清需求，避免「我以為你以為」

沒 SBE 的痛

  • PM 寫 PRD、Dev 猜需求、QA 寫測試各自為政
  • 實作到一半才發現「這 case 怎麼處理？」
  • 需求/測試文件雙寫、雙腐爛

有 SBE 後

  • 需求轉成例子：Given X, When Y, Then Z
  • 例子 = PRD = 驗收條件 = 測試骨架
  • 邊界條件先吵清楚再寫 code
你：/spec-by-example 紅利點數折抵

Claude:
  1. 讀 wiki/systems/、procedures/ 既有結帳知識
  2. 互動釐清邊界（折抵上限？與折扣碼同用？退貨？）
  3. 寫 Given/When/Then 規格到 specs/sbe/YYYY-MM-DD.md
  4. 與 wiki 既有頁面建立雙向 [[wikilink]]

→ 規格 = 驗收條件 = 測試骨架 = 活文件，四合一

SBE 不是新概念（BDD 業界用很多年），但「寫例子」是高摩擦動作——以前要開會、寫表、跨團隊對齊；Claude 讓這件事變便宜，所以才開始真的普及。 強調 specs/sbe/ 跟 wiki/ 雙向 wikilink，是「規格 ↔ 知識」可逆查詢。

安全護欄（再強調一次）

防線位置角色
1~/.claude/settings.json機械 deny / ask（17 + 17 條）
2產品 wiki CLAUDE.md 安全守則AI 主動自律
3README「安全注意事項與護欄」人類 SOP + 出事處置

機械擋 + AI 自律 + 人類把關，缺一不可。

勿信 AI 永不犯錯。三層並行是設計、不是冗餘。

這頁是「不能讓人聽完覺得 AI 萬能」的反例。一定要強調人類仍要把關。
