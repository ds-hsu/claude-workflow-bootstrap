# Specification by Example — 需求理解優先的開發入口

使用方式：`/spec-by-example <需求描述>` 或 `/sbe <需求描述>`

> **核心原則**：AI 不可同時掌控 spec 和 implementation。
> 在 Agentic 時代，SBE 的價值是「用具體例子暴露 AI 的理解落差」。
> 人類確認 spec 後才進入開發，避免 AI 自己批改自己的作業。

## 適用時機

- 收到新功能需求
- 收到修改現有行為的請求
- 需求描述模糊或有多種解讀
- 跨模組/跨服務的變更

**不適用**：純 bug fix（直接定位修復）、配置變更、文件更新

## 強制流程（不可跳過任何步驟）

### Step 1: 需求解析

將使用者的需求拆解為結構化描述：

```
需求原文：（使用者的原始描述）
我的理解：（用自己的話重述，暴露可能的理解偏差）
涉及角色：（哪些 actor 會與這個功能互動）
核心動詞：（這個功能要「做什麼」— 建立/修改/刪除/監控/通知...）
```

### Step 2: 假設審計 + 知識探查

查詢知識庫和 codebase，建立需求的完整上下文：

**Step 2-A：假設清單（Knowledge Anchoring，必做）**

根據 Step 1 的需求解析，列出此規格依賴的 3-7 個關鍵假設，逐一查 wiki 驗證：

| # | 假設內容 | Wiki 查詢（search/get_concept） | 狀態 |
|---|---------|-------------------------------|------|
| 1 | ...     | [wiki 頁面名稱]                 | ✅ / ⚠️ / ❌ |

狀態說明：
- ✅ 確認：wiki 有明確記載，假設正確，可繼續
- ⚠️ 待確認：wiki 無記載，假設來自推斷，標記為風險項
- ❌ 假設錯誤：wiki 與假設矛盾，**必須停止並回報使用者，不可繼續產出 spec**

**Step 2-B：Wiki MCP 影響範圍查詢**

1. **Wiki MCP 查詢**：
   - `search(需求關鍵字)` — 找相關概念和設計決策
   - `related(概念, depth=2)` — 找連鎖影響的模組
   - 列出所有「既有設計決策」和「修改注意事項」

2. **Codebase 探查**：
   - `grep` 搜尋現有相關實作（API routes、handlers、stores）
   - 列出會被影響的檔案和函式
   - 標記「需要同步修改」的位置

3. **輸出：系統邊界地圖**
   ```
   涉及模組：[列出]
   現有行為：[描述目前系統怎麼運作]
   影響範圍：[哪些地方會被改動]
   設計約束：[wiki 中記錄的不可違反原則]
   假設風險：[⚠️ 項目列表，待使用者確認]
   ```

### Step 3: Example Mapping

為需求產出 3 類具體例子，每個都用 Given/When/Then 格式：

#### Happy Path（正常流程）
至少 2 個例子，覆蓋最常見的使用情境。

```gherkin
# 例子 1: [情境名稱]
Given [前置條件 — 系統處於什麼狀態]
When  [觸發動作 — 使用者/系統做了什麼]
Then  [預期結果 — 應該發生什麼]
And   [副作用 — 其他系統應該怎麼反應]
```

#### Edge Cases（邊界情況）
至少 2 個例子，覆蓋邊界值和特殊情況。

```gherkin
# 例子: [邊界情境名稱]
Given [邊界前置條件]
When  [邊界觸發]
Then  [邊界處理方式]
```

#### Error Paths（錯誤處理）
至少 1 個例子，描述錯誤情況下的預期行為。

```gherkin
# 例子: [錯誤情境名稱]
Given [導致錯誤的前置條件]
When  [觸發動作]
Then  [錯誤應該怎麼被處理]
And   [使用者應該看到什麼回饋]
```

#### Contract Sketch（介面契約草案）
如果涉及 API 或跨服務通訊，附上 input/output 契約：

```
Endpoint: [METHOD /path]
Input:  { field: type, ... }
Output: { field: type, ... }
Error:  { code: number, message: string }
```

### Step 4: 人類確認 Gate（硬性關卡）

> **這是整個流程的核心。不可跳過，不可自動通過。**

將 Step 1-3 的完整產出呈現給使用者，明確要求確認：

```
════════════════════════════════════════
  SBE Specification Review
════════════════════════════════════════

[Step 1 需求解析結果]
[Step 2 系統邊界地圖]
[Step 3 Example Mapping 全部例子]

────────────────────────────────────────
  請確認以上規格：
  1. 例子是否正確反映你的需求？
  2. 有沒有遺漏的邊界情況？
  3. 錯誤處理方式是否符合預期？
  4. 介面契約是否合理？

  請回覆：
  - 「確認」→ 進入開發
  - 「修改 + 你的修正」→ 我更新 spec
  - 「新增 + 你的例子」→ 我加入新例子
════════════════════════════════════════
```

**等待使用者回覆，不可自行決定進入開發。**

**Revision Gate（計畫修改的強制路徑）**

使用者對 spec 不滿意時，修改前必須先執行：
1. 定位：「使用者不滿意的點，對應到 Step 2-A 的哪個假設？」
2. 重查：只針對該假設重新查 wiki（不需全量重跑 Step 2-A）
3. 修正：確認假設修正方向後，才進入 spec 修改
4. 更新：假設清單中的狀態必須更新，再呈現修訂後的 Step 4 review

不得跳過 Revision Gate 直接修改 spec，否則會重蹈同樣的理解錯誤。

### Step 5: Spec 持久化

使用者確認後，將最終 spec 寫入專案中：

1. 建立 `specs/sbe/YYYY-MM-DD-<feature-name>.md`
2. 內容包含完整的 Step 1-3 產出 + 使用者確認的修改
3. 這份文件在後續開發中作為 acceptance criteria 的唯一來源

### Step 6: 轉入開發

Spec 確認後，自動銜接開發流程：

1. 將確認的 Given/When/Then 例子轉化為測試骨架（但不寫實作）
2. 列出開發任務清單（基於 Step 2 的影響範圍）
3. 開始實作，每完成一個功能點就用 spec 中的例子驗證

**開發過程中如果發現 spec 未覆蓋的情況**：
- 停下來，補充新的例子
- 再次經過人類確認
- 然後繼續開發

## Telegram 模式

透過 Telegram 收到需求時，流程相同但格式更簡潔：

- Step 1-3 壓縮為一則結構化訊息
- Step 4 Gate 透過 Telegram 確認
- 使用者回覆「ok」「確認」「可以」等即視為通過

## 與現有流程的整合

```
舊流程：需求 → Phase 0 影響分析 → Phase 1 BDD → 實作
新流程：需求 → /sbe（整合 Phase 0 + 1 + 人類 Gate）→ 實作 → Phase 4
```

`/sbe` 取代了 `/impact-analysis` + `/test-scenario` 的組合，
並加入了 SBE 的核心價值：**具體例子 + 人類確認**。

## 反模式（永遠不要做）

1. **跳過 Step 4**：不管需求多簡單，都必須經過人類確認
2. **AI 自己確認自己的 spec**：這就是 TDD 對 AI 失效的原因
3. **例子太抽象**：Given/When/Then 必須包含具體值，不能用「某個值」
4. **只有 Happy Path**：Edge Case 和 Error Path 往往是 bug 的來源
5. **Spec 寫完就丟**：必須持久化到 `specs/sbe/`，開發中持續參照
6. **修改 spec 時跳過 Revision Gate**：不定位錯誤假設就直接改，會重蹈同樣的理解錯誤
7. **假設審計走過場**：沒有實際查 wiki、只列假設不驗證，等於沒做 Step 2-A
